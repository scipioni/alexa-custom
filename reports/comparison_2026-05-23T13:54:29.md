## Trend Analysis

### CPU — significant improvement
The hottest thread dropped from **86.8% → 30.8%** (-65%). This is directly attributable to removing the second KaldiRecognizer (`display_rec`) that was processing every audio chunk in parallel with `stage1`. Total process CPU fell from **123% → 76.5%** (-38%), taking the host from near-saturation (5.6% idle) to comfortable headroom (**54.9% idle**). The system is no longer CPU-bound.

### Memory — significant improvement  
RSS dropped from **372 MB → 270 MB** (-27%, -102 MB). The KaldiRecognizer object that was freed held roughly 100 MB of Kaldi decoder state (acoustic model cache + lattice buffers). Private dirty heap fell by **101 MB** (331 → 230 MB), confirming the reduction is real heap, not just virtual mapping. No memory growth is visible at this uptime — this should be confirmed at 30+ minutes.

### Temperature — improved
Peak CPU temperature fell from **61.0 °C → 55.1 °C** (-6 °C). Expected consequence of the CPU reduction. The SoC is no longer running at thermal ceiling.

### Memory leak watch
Both snapshots show 0 swap for the process itself. Snapshot 2 uptime is only 31s — insufficient to assess leak tendency. The next comparison (after ≥30 min uptime) should focus on whether RSS drifts upward from its current 270 MB baseline.

### WebSocket connections
The prune loop change has not yet had time to act (30s interval, process is 31s old). If the 20 connections don't shrink after the first prune cycle, it means all connections are genuinely alive (multiple browser tabs). No action needed in that case.

## Summary

The optimization pass delivered measurable real-world gains within the same session:

- **−38% CPU** (process), **−65%** on the single hottest thread
- **−27% RSS** (−102 MB heap freed from second Kaldi decoder)
- **−6 °C** peak SoC temperature
- Host went from 5.6% idle (near-saturated) → **54.9% idle**

The primary driver was fix #1 (dropping `display_rec`). The other fixes (dispatch loop reuse, AudioWatcher debounce, RMS gating) reduce micro-overhead and will show more clearly under sustained load.
